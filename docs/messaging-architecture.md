# Dharmic Data messaging architecture

## The short version

Dharmic Data is Jai Bhagat's public record of making AI useful for ordinary people and community work. Jai has spent more than ten years across software and teaching. He has taught economic empowerment work for free for years. Paid one-to-one sessions now fund the time to publish working demos, document what succeeds and fails, and learn what a future nonprofit collective should become.

Dharmic Data is not a nonprofit entity today. The stated aim is to seed and document the collective through June 2027.

## The audience

### The working person with scattered AI experiments

- Current reality: "I keep trying AI tools, but I still do not have one workflow I trust."
- Current alternatives: tutorials, prompt lists, more subscriptions, and broad courses.
- Why those fail: they start with a tool instead of a recurring problem in the person's own work.
- What is at risk: more time, more subscriptions, and no lasting change in how the work gets done.
- Promise: choose one useful workflow, decide which tool fits it, set the data boundary, and leave with a 30 day test.
- Proof: public demos, traces, source records, and a written plan from the session.
- Primary action: book AI guidance.

### The builder working across personal files or a repository

- Current reality: "I can make a demo, but I do not know if it is safe, useful, or worth maintaining."
- Current alternatives: copy a reference app, put a model in every step, or buy more capacity before testing the job.
- Why those fail: the demo is not tested against real data, failure cases, or the limits of the machine.
- What is at risk: private data, wasted build time, and a system that looks convincing without being dependable.
- Promise: define the smallest useful test, keep the data and action boundaries visible, and measure the result before expanding it.
- Proof: Shakti's public and local paths, evaluation records, source receipts, and published postmortems.
- Primary action: book AI guidance.

### The educator or community advocate

- Current reality: "I want to use AI in service of people, but I do not want to bring a vague or extractive project into my community."
- Current alternatives: generic workshops, closed vendor demos, or a public launch without a real-world feedback loop.
- Why those fail: people are asked to trust the technology before they can inspect the sources, limits, or next step.
- What is at risk: community trust and time that could have gone to a simpler public service.
- Promise: begin with the public need, use AI only where it adds value, and publish the proof people need to judge the work.
- Proof: Shakti keeps AI out of its public critical path and shows every data source and processing step.
- Primary action: try Shakti, then book guidance for a related project.

## The difference

The usual offer starts with a model, a feature list, or a promise of automation. Dharmic Data starts with a person's real work.

1. Name the recurring problem before choosing a tool.
2. Use the person's own files, repository, or public data when it is safe to do so.
3. Decide what the model may read and which actions always need review.
4. Test one useful task before expanding the system.
5. Publish evidence, limits, and failed attempts when the work is public.
6. Decline AI when ordinary code or a human decision is the better choice.

The lookup stays a focused civic tool. It may link to the author page, but the record search remains the main job.

## Copy rules

- Say "working toward a nonprofit collective" until an entity exists.
- Show proof before asking for a booking.
- Use one main action per page.
- Preview the artifact the person will leave with.
- State what the session does not include.
- Mark future offers as tests, not available products.
- Use plain words. Avoid claims about transformation that the evidence cannot support.
